## 1. 화폐 운영 규칙

### 1.1 기본값 (교사가 학급별 조정 가능)

| 항목 | 기본값 | 범위 | 비고 |
|---|---|---|---|
| 화폐 단위 | 500원 | 100~1,000원 | 너무 작으면 동기 저하, 너무 크면 가치 희석 |
| 학기 초 시작 잔액 | 1,000원 | 0~5,000원 | 0원 시작은 동기 저하 위험 |
| **최저잔액 보호** | **500원** | **변경 불가** | **반응대가 차감 하한선, 시스템 강제** |
| 주간 이자율 | 0.5% | 0~2% | 최소잔액 이상 보유 시 적용 |
| 이자 적용 최소잔액 | 2,000원 | 교사 설정 | 저축 장려용 |

### 1.2 잔액 이월 정책
- 학기 말 잔액 이월 여부: **교사 결정** (`system_settings.balance_carryover`)
- 이월 시: 다음 학기 시작 잔액에 합산
- 미이월 시: 학기 말 통장 출력 후 잔액 초기화

### 1.3 인플레이션 방지
- 가게 물가 조정 권한: **교사 전담**
- 권장 점검 주기: 월 1회
- 기준: 학생 평균 주간 수입 대비 가게 아이템 가격 균형 유지

## 2. 접근 및 인증 정책

### 2.1 학급 식별코드

```
형식: [학교약어]-[연도]-[순번]
예시: NDG-2026-001 (남대구초 2026년 첫 번째 학급)
발급: 학급 개설 시 서버 자동 생성
관리: 교사 보관 (분실 시 재발급 가능)
유효기간: 학년도 말 자동 만료 (is_active = false)
```

### 2.2 사용자별 인증 방식

| 사용자 | 인증 방법 | PIN 재설정 |
|---|---|---|
| 교사 | 학급코드 + PIN 4자리 | 교사 직접 변경 |
| 학생 | 학급코드 + 이름 선택 + PIN 4자리 | 교사가 관리 페이지에서 초기화 |
| ATM | QR카드 스캔 (PIN 없음) | QR 재출력 |

### 2.3 학부모 접근
- **완전 배제** (통장 열람, 알림 등 일체 없음)
- 소통 채널: 종이 통장 출력물만

### 2.4 PIN 보안
- PIN은 **bcrypt 해시** 저장 (평문 저장 절대 금지)
- 학생 PIN 분실 시: 교사가 관리 페이지에서 새 PIN으로 초기화
- 교사도 학생 원래 PIN 확인 불가 (해시만 저장)

## 3. 주식 운영 정책

### 3.1 날씨 연동 종목 (자동)
- 매일 오전 08:30 날씨 API 호출 (학급별 도시 설정 가능, 기본 대구)
- 주가 자동 계산·갱신 (계산 공식: `07_자동화스케줄러_날씨API.md` §5.2 참조)
- 학생이 날씨를 예측하고 투자 → 시장경제 교육 효과

### 3.2 교사 커스텀 종목 (수동)

**운영 원칙:**
- 종목 이름: **학생들과 함께 정하기** (수업 도입 활동 — 소유감 부여)
- 주가 조정: 교사 PC에서 언제든 가능 (조정 유형 6가지)
- 학생은 조작 사실 알지 못함 → 자연스러운 시장 원리 경험
- `teacher_note`에 조정 이유 기록 (학생에게 절대 비공개)

**주가 조정 활용 시나리오:**
```
상황 A — 잔액 불균형 조정
특정 학생 많이 보유 종목 폭등 → 잔액 격차 자연스럽게 해소

상황 B — 학급 이벤트 연계
소풍 전날 "소풍주식" 급등 → 기대감 + 투자 경험 연결

상황 C — 경제 교육 시나리오
의도적 폭락 → "손실"과 "위험" 개념 교육

상황 D — 동기 부여
오래 보유한 학생 보상 차원 완만한 상승 연출
```

**조정 타입별 배율 (서버 처리):**
```
surge:  ×2~3배 (랜덤)
rise:   ×1.2~1.5배 (랜덤)
flat:   ×1 (유지)
fall:   ×0.7~0.8배 (랜덤)
crash:  ×0.3~0.5배 (랜덤)
manual: 교사 직접 입력값
```

### 3.3 이자 정책
- 조건: `system_settings.interest_min_balance` 이상 보유 시
- 주기: 금요일 14:00 자동 지급
- 목적: "돈을 묵혀두면 늘어난다" 저축 개념 체험
- 통장 기록: "저축이자 +○○원"

## 4. 데이터 보관 및 삭제 정책

### 4.1 보관 기간
- 학생 거래·행동 데이터: **IEP 수립 기간과 동일** (`system_settings.data_retention_months`)
- 일반적으로 해당 학년도 + 1년 (기본값 12개월)

### 4.2 삭제 방식
- 보관 기간 만료 시 자동 삭제 **예고 알림** (교사에게 D-30)
- 교사 확인 후 삭제 실행 (자동 즉시 삭제 아님)
- 삭제 전 학기말 종합 통장 출력 강력 권장

### 4.3 학생 졸업·전학 시
- 교사가 수동으로 해당 학생 비활성화 (`students.is_active = false`)
- 데이터 보관 기간까지 DB 유지 (IEP 자료 보존)

## 5. 보호자 동의서 정책

### 5.1 동의서 유형 및 필수 여부

| 유형 | 시점 | 필수 여부 |
|---|---|---|
| 시스템 전반 동의서 | 학기 초 | 권장 |
| **반응대가 적용 동의서** | **반응대가 활성화 전** | **필수** |
| 행동계약 보호자 확인서 | 계약 체결 시 | 권장 |
| 데이터 보관 동의서 | 학기 초 | 권장 |

### 5.2 제공 방식
- `consent_templates` 테이블에 저장된 템플릿 PDF 출력
- 종이 서명 후 교사 보관
- 시스템 내 서명 여부 체크박스 (문서 업로드 없이)

## 6. 반응대가 적용 정책

### 6.1 필수 조건 (모두 충족 시만 활성화)
1. **FBA(기능행동평가) 완료** → `fba_records`에 기록
2. **보호자 서면 동의** → `students.parental_consent_rc = true`
3. **풍부한 강화 기회 병행 확인** → 교사 체크

### 6.2 시스템 강제 규칙
- 최저잔액 **500원 이하 차감 불가** (서버 레벨 강제 — 클라이언트 우회 불가)
- 기본값: **모든 학생 비활성** (`students.response_cost_enabled = false`)
- 활성화 시 통장에 투명하게 기록

### 6.3 모니터링
- 반응대가 적용 후 2주 이내 행동 변화 확인
- 효과 없거나 행동 증가 시 즉각 중단

## 7. 배포 정책

### 7.1 배포 형태
- 무료 배포 (학운위 심의 별도 없음)
- 도메인: 미정 (iepon.site 서브도메인 또는 별도)

### 7.2 멀티학급 격리 구조
```
학교A 특수교사 → 학급 개설 → NDG-2026-001
학교B 특수교사 → 학급 개설 → JIN-2026-001
→ 학급별 완전히 격리된 독립 공간
→ 데이터 상호 접근 불가 (RLS 강제)
```

### 7.3 방학 처리
- 교사가 "방학 모드" 활성화 → `class_codes.is_active = false`
- 모든 Cron Job 해당 학급 스킵 (불필요한 정산 방지)
